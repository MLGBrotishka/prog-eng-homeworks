# Домашнее задание 01: Документирование архитектуры в Structurizr

## Вариант 16 — Система заказа такси (аналог Uber)

### Описание системы
Система предназначена для взаимодействия между пассажирами и водителями. Пассажиры заказывают поездки, водители принимают и выполняют их. Система отслеживает состояние поездок, хранит историю и управляет данными пользователей и водителей.

---

### Данные предметной области
| Сущность | Описание |
|----------|----------|
| **Пользователь (User)** | Пассажир, заказывающий поездки. Атрибуты: логин, email, телефон, имя, фамилия |
| **Водитель (Driver)** | Водитель такси, выполняющий заказы. Атрибуты: ФИО, номер авто, лицензия, рейтинг |
| **Поездка (Ride)** | Заказ на перевозку от точки A до точки B. Атрибуты: маршрут, стоимость, статус, время |

---

### API (все варианты использования)
| # | Операция | HTTP метод | Endpoint |
|---|----------|-----------|----------|
| 1 | Создание нового пользователя | `POST` | `/api/v1/users` |
| 2 | Поиск пользователя по логину | `GET` | `/api/v1/users?login={login}` |
| 3 | Поиск пользователя по маске имя и фамилии | `GET` | `/api/v1/users?name={mask}` |
| 4 | Регистрация водителя | `POST` | `/api/v1/drivers` |
| 5 | Создание заказа поездки | `POST` | `/api/v1/rides` |
| 6 | Получение активных заказов | `GET` | `/api/v1/rides?status=active` |
| 7 | Принятие заказа водителем | `PATCH` | `/api/v1/rides/{id}/accept` |
| 8 | Получение истории поездок пользователя | `GET` | `/api/v1/users/{id}/rides` |
| 9 | Завершение поездки | `PATCH` | `/api/v1/rides/{id}/complete` |

---

## Архитектура

### Роли пользователей
1. **Пассажир** — заказывает поездки, оплачивает, просматривает историю
2. **Водитель** — регистрируется, принимает и завершает заказы, получает уведомления

### Внешние системы
| Система | Назначение |
|---------|-----------|
| **Картографический сервис** | Маршрутизация и геоданные (Yandex Maps / Google Maps) |
| **Платежная система** | Обработка платежей: холдирование и списание средств |
| **Push-сервис** | Push-уведомления водителям и пассажирам (FCM / APNs) |

### Контейнеры системы

#### Клиентские приложения
| Контейнер | Технология | Ответственность |

#### Бизнес-сервисы

|-----------|-----------|----------------|
| User Service | C++, Userver | Регистрация пользователей, аутентификация, поиск по логину и маске |

#### Хранилища данных
| Контейнер | Технология | Ответственность |

### Dynamic-диаграммы

#### UC01: Создание заказа, бронирование и принятие поездки
Основной сценарий жизненного цикла заказа:

1. Пассажир указывает адреса в приложении
2. Gateway запрашивает расчёт маршрута у картографического сервиса
3. Пассажиру возвращается стоимость и ETA
4. Пассажир подтверждает заказ
5. Ride Service создаёт заказ (статус `PENDING`)
6. Gateway запрашивает холдирование средств в платежной системе
7. Платёжная система подтверждает холд (webhook)
8. Ride Service обновляет статус (`BOOKED`)
9. Ride Service публикует событие `ride.requested` в Kafka
10. Push-уведомление доставляется водителям
11. Водитель принимает заказ
12. Ride Service назначает водителя, обновляет статус (`ACCEPTED`)
13. Публикуется событие `ride.accepted`

#### UC02: Завершение поездки и списание оплаты
1. Водитель нажимает «Завершить поездку»
2. Ride Service обновляет статус (`COMPLETED`)
3. Gateway запрашивает списание забронированных средств
4. Платёжная система подтверждает списание
5. Публикуется событие `ride.completed`
6. Аналитические данные записываются в MongoDB
7. Пассажир получает push-уведомление о завершении

# Домашнее задание 02: Разработка REST API сервиса

## Реализация
Микросервисы реализованы на языке **C++** с использованием современного асинхронного фреймворка **Userver**.

### Основные возможности
- **JWT Аутентификация:** Защита эндпоинтов с использованием токенов.
- **Микросервисная архитектура:** Разделение на User, Driver и Ride сервисы.
- **База данных:** Использование SQLite для хранения данных пользователей, водителей и поездок.
- **Docker:** Полная контейнеризация всех сервисов и Nginx в качестве API Gateway.
- **Документация:** Интерактивный Swagger UI для тестирования API.

# Домашнее задание 03: Проектирование и оптимизация БД PostgreSQL

## Реализация
В рамках данного задания система была переведена с SQLite на **PostgreSQL**.

### Основные возможности
- **Миграция БД:** Полный переход на PostgreSQL с использованием компонента `userver::components::Postgres`.
- **Схема и данные:** Созданы скрипты `services/schema.sql` (со структурами таблиц и индексами) и `services/data.sql` (с тестовыми данными для пользователей, водителей и поездок).
- **Справочник запросов:** В `services/queries.sql` собраны все SQL-запросы для 9 основных операций API с использованием плейсхолдеров `$1, $2`.
- **Оптимизация:** Добавлен файл `services/optimization.md` с описанием схемы, обоснованием выбора индексов, анализом планов выполнения запросов (EXPLAIN) и стратегией партиционирования таблицы поездок.

### Как запустить с PostgreSQL
1.  **Сборка и запуск через Docker Compose:**
    ```bash
    make build
    ```
    *Поднимется база данных PostgreSQL, выполнится инициализация схемы и данных, после чего запустятся микросервисы (User, Driver, Ride) и Nginx.*

2.  **Проверка работы API (Swagger):**
    Откройте в браузере: http://localhost:8080/swagger/index.html

3.  **Документация по оптимизации:**
    См. файл `services/optimization.md` для деталей по индексам, планам выполнения и структуре БД.

# Домашнее задание 04: Проектирование и работа с MongoDB

## Реализация
Система расширена для поддержки работы с NoSQL базой данных MongoDB. В рамках этого задания микросервис **User Service** (реализованный на C++ `userver`) был полностью переведен с PostgreSQL на MongoDB.

### Основные возможности
- **Интеграция в C++:** `User Service` напрямую работает с MongoDB, используя нативный компонент `userver::components::Mongo`. 
- **Адаптация ID:** Для обеспечения совместимости со старыми SQL-таблицами `drivers` и `rides` генерация идентификатора реализована через числовой `int64_t`, записываемый в поле `_id` документа MongoDB.
- **Изоляция БД:** Связи `FOREIGN KEY` к пользователям в PostgreSQL удалены, так как базы данных микросервисов теперь физически изолированы (User → MongoDB, Driver/Ride → PostgreSQL).
- **Запросы и Данные:** В файле `services/queries.js` представлены необходимые CRUD операции и сложные агрегации. Схемы валидации и тестовые данные загружаются при инициализации базы.

### Как проверить работу с MongoDB
1. **Сборка и запуск:**
	```bash
    make build
	```
	*Поднимется кластер из PostgreSQL, MongoDB и C++ сервисов.*

2. **Запуск тестов:**
	```bash
    make test
	```
	*Интеграционные тесты для User Service будут прозрачно выполняться через MongoDB.*

3. **Ручная проверка запросов MongoDB:**
	Вы можете подключиться к контейнеру:
	```bash
    docker exec -it taxi-mongodb mongosh -u taxi_mongo -p taxi_mongo_password --authenticationDatabase admin taxi_db
	```
	И запустить демо-скрипт с агрегациями:
	```js
    load('/docker-entrypoint-initdb.d/queries.js');
	```

# Домашнее задание 05: Оптимизация производительности (Вариант 16)

В рамках ДЗ 05 в систему интегрирован Redis для кеширования и Rate Limiting:

### 1. Кеширование (Cache-Aside)
* **`GET /api/v1/rides?status=active`** (ride-service) — кешируется на 10 секунд (ключ `rides:active`). Инвалидируется явно при `PATCH /api/v1/rides/{id}/accept` и `complete`.
* **`GET /api/v1/users/by-login`** (user-service) — профиль кешируется на 5 минут (ключ `user:login:{login}`).

### 2. Rate Limiting (Sliding Window)
Внедрен алгоритм Sliding Window Counter с атомарными `INCR` + `EXPIRE` через Redis. Возвращает статус `429 Too Many Requests` и заголовки `X-RateLimit-*`.
* **`POST /api/v1/auth/login`**: 5 запросов в минуту (по IP).
* **`POST /api/v1/rides`**: 10 запросов в минуту (по passenger_id).
* **`GET /api/v1/rides?status=active`**: 30 запросов в минуту (по IP).

Подробная архитектура описана в [performance_design.md](services/performance_design.md).

**Запуск:**
```bash
make build
```
Контейнер `taxi-redis` запускается автоматически вместе с остальными БД.
