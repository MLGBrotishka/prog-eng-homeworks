# Архитектура оптимизаций: Кеширование и Rate Limiting

## 1. Анализ производительности и Hot Paths

В ходе профилирования системы были выявлены следующие проблемные места:
- **`GET /api/v1/rides?status=active` (ride-service)** — Высокочастотный запрос, используемый водителями для поллинга доступных поездок. Требует полного сканирования таблицы или фильтрации по неиндексированному статусу в PostgreSQL.
- **`GET /api/v1/users/by-login?login=...` (user-service)** — Запрос профиля пользователя при аутентификации. Частота запросов может резко возрастать, а данные профиля меняются редко.
- **`POST /api/v1/rides` и `POST /api/v1/auth/login`** — Уязвимы к флуду заказами и брутфорс-атакам соответственно.

## 2. Стратегия кеширования

Для снижения нагрузки на БД внедрен Redis в роли распределенного кеша.
Используется паттерн **Cache-Aside** (чтение с промахом обращается к БД и сохраняет результат в кеш).

| Endpoint | Данные | Ключ в Redis | TTL | Стратегия инвалидации |
|---|---|---|---|---|
| `GET /rides?status=active` | Список JSON-объектов активных поездок | `rides:active` | 10 сек | **Явная инвалидация** (удаление ключа) при любом изменении статуса поездки (`AcceptRide`, `CompleteRide`) |
| `GET /users/by-login?login={login}` | JSON профиль пользователя | `user:login:{login}` | 5 мин | Неявная инвалидация по истечению TTL. Данные обновляются редко, stale read в 5 мин допустим |

## 3. Стратегия Rate Limiting

Для защиты от перегрузки и злоупотреблений реализован алгоритм **Sliding Window Counter** на базе Redis.

**Алгоритм:**
1. Окно вычисляется как `unix_ts / window_size`.
2. Ключ в Redis: `rl:{endpoint}:{ip/user_id}:{window}`.
3. Используется атомарная операция `INCR` для счетчика.
4. При первом инкременте (`count == 1`) устанавливается `EXPIRE`.

| Endpoint | Ключ ограничения | Лимит | Окно | Поведение при превышении |
|---|---|---|---|---|
| `POST /auth/login` | Client IP | 5 req | 60s | `429 Too Many Requests` |
| `POST /rides` | Passenger ID | 10 req | 60s | `429 Too Many Requests` |
| `GET /rides?status=active` | Client IP | 30 req | 60s | `429 Too Many Requests` |

**Заголовки Rate Limiting:**
Сервисы возвращают информационные заголовки:
- `X-RateLimit-Limit` — лимит запросов в окне.
- `X-RateLimit-Remaining` — оставшееся количество запросов.
- `X-RateLimit-Reset` — UNIX timestamp, когда окно будет сброшено.

## 4. Метрики эффективности

Для мониторинга эффективности введенных решений используются стандартные метрики Redis (`INFO STATS`):
- **Cache Hit Rate** = `keyspace_hits / (keyspace_hits + keyspace_misses)`. Ожидаемый hit rate для активных поездок > 90%.
- **Latency (p50, p99)** для защищенных endpoint-ов должна снизиться за счет ответов из RAM (Redis) вместо Disk/B-Tree (PG/Mongo).
- Снижение CPU Usage на узлах PostgreSQL и MongoDB.

## 5. Влияние на систему

- **Снижение нагрузки на БД**: Устранение 90% запросов к `ListActiveRides` освобождает worker-ы PostgreSQL.
- **Защита от перегрузки**: Rate Limiter отсекает флуд на раннем этапе (в handler-е до обращения к БД или создания сессий).
- **Отказоустойчивость**: Rate Limiter реализует паттерн *fail-open* — при недоступности Redis запросы пропускаются, чтобы не блокировать работу сервиса при сбое инфраструктуры мониторинга.
