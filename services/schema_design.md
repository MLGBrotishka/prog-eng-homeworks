# Проектирование схемы MongoDB — Пользователи (Вариант 16)

В рамках данной работы в MongoDB перенесена сущность **Пользователи (Users)**. Остальные сущности системы (Водители и Поездки) остаются в реляционном хранилище (PostgreSQL). Такое разделение позволяет использовать преимущества документной модели для профилей пользователей (гибкие настройки, метаданные), сохраняя строгую транзакционность для заказов.

## Коллекция: `users` (Пользователи)

Хранит профили пассажиров. Нет подсущностей, требующих выделения в отдельную коллекцию.

```json
{
  "_id": Int64,             // ручное управление ID (синхронизировано с SQL-совместимыми сервисами)
  "login": String,          // уникальное поле, проиндексировано
  "email": String,          // уникальное поле, проиндексировано
  "password_hash": String,
  "first_name": String,
  "last_name": String,
  "phone": String,
  "created_at": Date,
  "is_active": Boolean,
  "preferences": {          // embedded (вложенный документ): небольшой размер, специфично для пользователя
    "language": String,
    "notifications_enabled": Boolean,
    "preferred_car_class": String
  },
  "rating": {               // embedded: агрегированные данные о поведении пользователя
    "average": Double,
    "count": Int
  }
}
```

**Обоснование Embedded vs References:**
- `preferences` (Настройки) — **embedded**: настройки всегда извлекаются вместе с данными пользователя, не шарятся между сущностями, занимают мало места (<100 байт).
- `rating` (Рейтинг) — **embedded**: денормализованная агрегация, обновляющаяся после каждой поездки. Вынесение в отдельную коллекцию потребовало бы дополнительных запросов при загрузке профиля.

## Интероперабельность и типы ID

Для обеспечения бесшовной интеграции между MongoDB и PostgreSQL:
1. **Тип идентификатора**: Поле `_id` в коллекции `users` имеет тип `Int64` (BSON Long). Это позволяет использовать ID пользователя как внешний ключ (BIGINT) в таблицах `drivers` и `rides` в PostgreSQL без необходимости преобразования типов.
2. **Генерация ID**: Идентификаторы генерируются на стороне сервиса (на основе UNIX timestamp в микросекундах) при создании нового пользователя.

## Стратегия индексирования

```javascript
users:     { login: 1 } unique
users:     { email: 1 } unique
users:     { first_name: 1, last_name: 1 }   // составной индекс для поиска по маске
```
