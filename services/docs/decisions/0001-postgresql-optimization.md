# 0001. Оптимизация базы данных PostgreSQL для системы такси

## Дата
2026-03-29

## Статус
Принято (Accepted)

## Контекст
Система такси обрабатывает большие объемы транзакций для пользователей, водителей и истории поездок. Нам необходима архитектура базы данных, которая обеспечит производительность O(log N) для операций поиска, оптимизирует сортировку для запросов, чувствительных ко времени, и позволит эффективно управлять жизненным циклом данных (архивирование старых поездок) без ущерба для доступности системы.

## Решение
В базе данных PostgreSQL реализованы следующие стратегии оптимизации:

### 1. Реляционная схема и ограничения
Спроектированы три нормализованные таблицы с соблюдением ссылочной целостности:
*   **`users`**: Единая таблица для всех сущностей с уникальными полями `login` и `email`.
*   **`drivers`**: Расширяет профили пользователей (отношение 1:1) с отслеживанием статуса.
*   **`rides`**: Основная транзакционная таблица, связывающая пассажиров, водителей, а также пространственные и финансовые данные.

### 2. Каталог стратегических индексов
Внедрены B-Tree индексы, ориентированные на конкретные операции API:
*   **Оптимизация поиска:** `idx_users_login` и `idx_users_name` для поиска пользователей.
*   **Производительность соединений (JOIN):** `idx_drivers_user_id` и `idx_rides_passenger_id/driver_id` для реляционных поисков.
*   **Фильтрация состояний:** `idx_drivers_status` и `idx_rides_status` для высокочастотной фильтрации активных сущностей.
*   **Производительность сортировки:** `idx_rides_created_at` для получения предварительно отсортированных результатов истории и активных списков, что исключает дорогостоящую сортировку в оперативной памяти.

### 3. Горизонтальная масштабируемость (Партиционирование)
Подготовлено решение для роста данных в таблице `rides` путем внедрения стратегии **RANGE Partitioning** по столбцу `created_at` (с месячными интервалами).

## Последствия
*   **Производительность:** Радикальное снижение стоимости запросов для исторических данных и операций поиска (переход от последовательного сканирования к сканированию по индексу).
*   **Управляемость:** Упрощенное архивирование путем удаления разделов (DROP TABLE) вместо ресурсоемких операций `DELETE`.
*   **Хранение:** Увеличение использования дискового пространства (примерно на 15-20% накладных расходов) из-за обслуживания индексов.
*   **Задержка:** Небольшие накладные расходы на операции `INSERT` из-за обновления индексов.
